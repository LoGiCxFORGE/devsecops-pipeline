Pre-commit (Local Security Layer)
What is pre-commit?

pre-commit

👉 A tool that runs checks before git commit

Why use it?
catches issues early (before pushing)
prevents bad code entering repo
faster feedback than CI
Flow
Write code → git commit → pre-commit runs → allow/block commit
When to use pre-commit run --all-files?

Use only when:

first setup
debugging
after changing config

👉 NOT needed for daily commits

Normal workflow
git add .
git commit -m "message"

👉 Hooks run automatically

Bypass (for testing vulnerabilities)
git commit --no-verify

👉 skips hooks

2️⃣ Linter vs Formatter
Linter (Ruff-check)

Ruff

👉 checks code quality and logic

Finds:
unused imports
bad patterns
potential bugs
Formatter (Ruff-format)

👉 fixes code style automatically

Fixes:
spacing
indentation
formatting
Difference
Tool	Purpose
Linter	detect problems
Formatter	fix appearance
Why --fix?
args: [--fix]

👉 automatically fixes issues instead of just reporting

3️⃣ Bandit & SAST
What is Bandit?

Bandit

👉 scans Python code for security issues

What is SAST?

👉 Static Application Security Testing

Meaning:
scans code without running it
detects vulnerabilities early
Example (your app)
query = f"SELECT * FROM users WHERE username = '{username}'"

👉 Bandit detects:

SQL Injection (CWE-89)
Key insight
SAST is pattern-based
not always 100% accurate
4️⃣ Gitleaks & Secret Scanning
What is Gitleaks?

Gitleaks

👉 detects secrets in code

What it finds
API keys
tokens
passwords
Why your first key didn’t trigger
# API_KEY = "sk_test_123456789"

👉 reasons:

commented
not matching strong pattern
Working example
API_KEY = "ghp_1234567890abcdefghijklmnopqrstuvwxyz"

👉 detected successfully

Key insight
secret scanners use patterns + entropy
not all strings are flagged
